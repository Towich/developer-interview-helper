# Техническое задание: приложение подготовки к собеседованиям (AchLine / CMP)

## 1. Назначение и платформы

**Назначение:** однопользовательское офлайн-приложение для повторения **теоретических** вопросов по стеку Android / Kotlin / смежным технологиям (Gradle и т.д.). Фокус первой версии — карточки с вопросом и ответом, без кодовых задач и без сети.

**Платформы:** Android и iOS на **Compose Multiplatform** (общий UI и по возможности общая бизнес-логика в `commonMain`).

**Язык:** интерфейс и тексты вопросов/ответов — **русский** (исходные материалы и структура тем предполагаются в папке `android-interview` на рабочем столе разработчика).

---

## 2. Исходные материалы и соответствие структуры

- Каталог **`android-interview`** (рабочий стол): оттуда при наполнении релиза берутся **иерархия технологий, категорий, тем** и краткие описания.
- В репозитории контент пакуется в **read-only** ресурсы приложения (например `composeApp/src/commonMain/composeResources/files/...` или согласованный путь внутри модуля).
- **Соответствие путей:** логическая иерархия в приложении должна совпадать с раскладкой по папкам при импорте, например:

  `Technology / Category / theme_slug.json`

  Точное соглашение об именовании файлов и slug'ов фиксируется при первом импорте (латиница в путях ресурсов, человекочитаемые названия — в JSON).

---

## 3. Модель данных и файловая структура контента

### 3.1. Иерархия

**Технология** → **Категория** → **Тема**.

- Одна **тема** = один JSON-файл.
- JSON-файлы сгруппированы по папкам **технологии** (и при необходимости вложенным папкам категории — как удобнее для ресурсов CMP).

### 3.2. Содержимое JSON одной темы (бандл)

Метаданные темы **не дублируются** в каждом вопросе: они задаются **контекстом файла** (путь + заголовок файла/корневой объект).

Рекомендуемая схема корня файла:

| Поле | Тип | Описание |
|------|-----|----------|
| `schemaVersion` | number | Версия схемы файла темы |
| `technologyId` | string | Идентификатор технологии (совпадает с папкой/реестром) |
| `categoryId` | string | Идентификатор категории |
| `themeId` | string | Уникальный id темы внутри категории |
| `themeTitle` | string | Отображаемое название темы |
| `questions` | array | Список вопросов |

Элемент `questions[]`:

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | Стабильный уникальный id (UUID или префикс + slug); **не меняется** между версиями приложения |
| `questionText` | string | Текст вопроса |
| `answerText` | string | Текст ответа (развёрнутый) |

### 3.3. Прогресс и пользовательские изменения (user overlay)

В пользовательском хранилище (отдельный JSON, см. раздел 4) хранятся **только дельты**, не полная копия всех вопросов:

- счётчики по `questionId`;
- список **добавленных** вопросов (с полными полями и привязкой к technology / category / theme);
- список **удалённых** `id` (soft-delete относительно бандла);
- версия схемы оверлея.

**Счётчики (минимум):**

| Поле | Описание |
|------|----------|
| `correctCount` | Число свайпов **вправо** («смог ответить»). Основной критерий выбора вопроса внутри темы. |
| `shownCount` | Сколько раз карточка с этим вопросом **показывалась** пользователю (см. раздел 5 — когда инкрементируется). Нужен для **ничьих** при одинаковом `correctCount`. |

Опционально для аналитики: `lastShownAt` (timestamp) — не обязателен для v1.

## 4. Хранение: бандл + пользовательский JSON

### 4.1. Read-only бандл

- Исходные JSON тем встроены в приложение как ресурсы; в рантайме **не перезаписываются**.

### 4.2. Запись пользовательских данных

- Файл (или несколько файлов) в **app-specific** каталоге:
  - **Android:** `context.filesDir` (или подпапка);
  - **iOS:** `Application Support` (подпапка приложения).
- **Первый запуск:** если файла оверлея нет — создать с пустыми структурами (`progress: {}`, `addedQuestions: []`, `removedIds: []`, `overlaySchemaVersion`).
- **Слияние при чтении:** итоговая модель = вопросы из бандла (по всем темам) **плюс** добавленные из оверлея **минус** `removedIds` **плюс** наложенные счётчики прогресса по ключу `questionId`.
- **Обновление приложения** с новыми JSON в бандле: новые `id` из бандла появляются в пуле; для существующих `id` **сохраняются** пользовательские счётчики и флаги удаления; конфликт одинаковых `id` из разных источников недопустим — id глобально уникальны.
- **Запись на диск:** атомарно (запись во временный файл → `rename`/replace), чтобы избежать порчи данных при сбое.
- **Миграции:** при росте `overlaySchemaVersion` предусмотреть простую миграцию полей (дефолты для новых ключей).

## 5. Поведение свайпов и обновление счётчиков

| Действие | `correctCount` | `shownCount` |
|----------|----------------|--------------|
| Карточка показана в вершине стека (стала верхней) | без изменений | +1 при первом появлении в качестве верхней в текущей «сессии показа» (см. уточнение ниже) |
| Свайп **вправо** («смог ответить») | +1 | без дополнительного инкремента за сам свайп, если показ уже учтён |
| Свайп **влево** («не смог») | без изменений | без дополнительного инкремента за свайп |

**Уточнение для v1:** `shownCount` увеличивается **один раз**, когда вопрос становится **верхней** карточкой в стеке (перед тем как пользователь свайпнет). Повторный показ того же вопроса позже снова увеличит `shownCount`. Так обеспечивается справедливый tie-breaker: среди вопросов с одинаковым `correctCount` приоритет у менее «засвеченных».

## 6. Доменная логика: выбор темы и вопроса

### 6.1. Метрика «неспрашиваемости» темы

Для каждой темы вычисляется:

`themeScore = сумма correctCount по всем активным (не удалённым) вопросам темы`.

**Выбор темы:** среди всех тем с хотя бы одним доступным вопросом выбирается тема с **минимальным** `themeScore`. При равенстве — тема с меньшей **суммой** `shownCount` по вопросам; затем лексикографически меньший `themeId`.

Интуиция: если по Retrofit ещё не было ни одного успешного ответа, а по Ktor уже были — сумма по Retrofit ниже, тема приоритетнее.

### 6.2. Выбор вопроса внутри темы

Среди вопросов выбранной темы (не удалённых):

1. Минимальный `correctCount`.
2. При равенстве — минимальный `shownCount`.
3. При равенстве — лексикографически меньший `id`.

### 6.3. Стек из трёх карточек

- На экране одновременно **до трёх** видимых карточек (верхняя + два нижних слоя с визуальным смещением/масштабом).
- Вопросы для второй и третьей карточки подбираются **той же политикой**, что и для первой, с условием **без повторов id** в текущем стеке.
- После свайпа верхней карточки стек дозаполняется следующим вопросом по правилам выше.

Если доступных вопросов меньше трёх (малая база), отображается столько, сколько есть.

## 7. UI/UX (Compose)

### 7.1. Экран стека

- Жесты: перетаскивание карточки влево/вправо с порогом срабатывания; опционально подсказки «не смог» / «смог».
- Три визуальных слоя; нижние слои слегка уменьшены и смещены (типичный паттерн «Tinder stack»).

### 7.2. Отображение карточки

- Виден текст вопроса; ответ — по тапу «Показать ответ» или раскрытие (конкретный паттерн — на усмотрение реализации, в ТЗ: **не показывать ответ до явного действия пользователя**).

### 7.3. FAB

- **Правый нижний FAB:** добавить вопрос. Открывается форма: выбор технологии, категории, темы (из существующих списков), поля текста вопроса и ответа; генерация нового уникального `id`. Сохранение — в user overlay (`addedQuestions`), без изменения файлов бандла.
- **Левый нижний FAB:** удалить **текущую верхнюю** карточку. Перед удалением — **диалог подтверждения**. После подтверждения: `id` вносится в `removedIds` (если вопрос из бандла) или удаляется из `addedQuestions` (если был добавлен пользователем). Стек пересобирается.

| Слой | Ответственность |
|------|-----------------|

## 10. Критерии приёмки (v1)

1. Приложение работает **офлайн** на Android и iOS.  
2. После перезапуска сохраняются **счётчики** и списки **добавленных/удалённых** вопросов.  
3. Новые вопросы из обновлённого бандла подхватываются без потери прогресса по старым `id`.  
4. Выбор темы и вопроса соответствует разделу 6 (включая tie-breaker по `shownCount` и `id`).  
5. Стек из **до трёх** карточек; свайп вправо увеличивает `correctCount`, влево — нет.  
6. FAB добавления и удаления (с подтверждением удаления) работают согласно разделу 7.

## 11. Принятые продуктовые решения (зафиксировано в ТЗ)

| Вопрос | Решение |
|--------|---------|
| Хранение прогресса и правок | Отдельный JSON-оверлей в app-specific storage поверх read-only бандла |
| Приоритет внутри темы | Минимальный `correctCount`; ничьи — по минимальному `shownCount`, затем по `id` |
| Приоритет между темами | Минимальная сумма `correctCount` по теме; ничьи — сумма `shownCount`, затем `themeId` |
| Удаление через левый FAB | Удаление **текущей верхней** карточки с **диалогом подтверждения** |
