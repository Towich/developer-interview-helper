# План: документ ТЗ (markdown)

## Цель

Создать файл спецификации (предлагаемое имя: [`docs/interview-app-tz.md`](docs/interview-app-tz.md) или [`ТЗ-interview.md`](ТЗ-interview.md) в корне репозитория — **уточнить при согласовании план**), который описывает продукт, данные, доменную логику, UI и нерешённые детали с рекомендациями.

## Исходные материалы

- Папка на рабочем столе `android-interview`: в ТЗ указать, что **структура технологий / категорий / тем** и краткие описания берутся оттуда при наполнении бандла; при необходимости приложить таблицу соответствия «папка на диске → путь в `composeResources`».

## Разделы документа ТЗ

### 1. Назначение и платформы

- Однопользовательское офлайн-приложение для теоретических вопросов (Android / iOS, Compose Multiplatform).
- Язык интерфейса и язык текстов вопросов — **зафиксировать в ТЗ** (предложение по умолчанию: RU, если материалы из `android-interview` на русском).

### 2. Модель контента и файловая структура

- Иерархия: **Технология** → **Категория** → **Тема**; один JSON-файл = **одна тема**, внутри — массив вопросов.
- Поля вопроса (минимум): стабильный `id` (UUID или строка), `questionText`, `answerText` (или ссылка на развёрнутый ответ — по вашему желанию), метаданные темы дублируются в файле или выводятся из пути (в ТЗ выбрать один вариант, чтобы не расходились данные).
- Поля прогресса (для оверлея): **`correctCount`** (число свайпов вправо) — основной критерий; опционально **`shownCount`** или **`wrongCount`** для аналитики и разрешения ничьих — **явно описать**: при выборе «только по успехам» логично хранить ещё «показы» или «всего свайпов», иначе у всех нерешённых вопросов `correctCount == 0` и нужен детерминированный tie-breaker (например меньше всего раз **показывали** карточку, затем случайный/стабильный порядок по `id`).

### 3. Хранение: бандл + пользовательский JSON (ваш выбор)

- **Read-only**: исходные JSON в ресурсах приложения (как в CMP принято: `composeResources` / expect/actual чтение файлов).
- **Запись**: отдельный файл (или несколько) в **app-specific directory** (Android `filesDir`, iOS `Application Support`): при первом запуске — **инициализация** (пустой оверлей или копия дефолтов); далее **слияние**: база из бандла + дельты из user JSON (добавленные вопросы, удалённые `id`, обновлённые счётчики).
- В ТЗ описать: атомарная запись (write temp → rename), миграция версии схемы JSON, что происходит при обновлении приложения с новыми вопросами в бандле (мерж по `id`, не затирать пользовательские счётчики).

### 4. Домен: выбор следующего вопроса

1. **Выбор темы**: «самая неспрашиваемая» — формализовать как минимум **агрегата по теме** от выбранной метрики (согласовано с примером Retrofit vs KTor: приоритет темы, где ещё не было успешных ответов / меньше всего их суммарно). Уточнить в ТЗ формулу: например `themeScore = sum(correctCount)` по всем вопросам темы; тема с минимальным `themeScore` выигрывает.
2. **Выбор вопроса внутри темы**: минимальный **`correctCount`**; при равенстве — tie-breaker из п.2 (например минимум показов, затем `id`).
3. **Стек из 3 карточек**: откуда брать 2–3 следующих вопроса после текущего — та же политика, без повторов в стеке; при исчерпании пула — циклически по тем же правилам.

### 5. UI/UX (Compose)

- Экран стека карточек в стиле Tinder: **3 видимых** слоя (масштаб/смещение/alpha по образцу типичного CardStack).
- Свайп **вправо** — «смог ответить» → инкремент успеха (и при необходимости показа).
- Свайп **влево** — «не смог» → **не** увеличивать успех; в ТЗ указать, увеличивается ли счётчик «показа» / «попытки» для tie-breaker.
- **FAB справа снизу** — добавление вопроса: диалог/экран с выбором технологии, категории, темы (и при необходимости создание новой темы = новый файл или запись в оверлей — зафиксировать).
- **FAB слева снизу** — удаление: в ТЗ указать явно **удаление текущей (верхней) карточки** или выбор из списка (по умолчанию — текущая карточка с подтверждением).

### 6. Слоистая архитектура (рекомендация для CMP)

- **data**: чтение бандла, чтение/запись user JSON, маппинг DTO → domain.
- **domain**: чистая логика выбора вопроса/темы, применение свайпа, add/remove.
- **presentation**: Compose, ViewModel (или аналог), состояние стека.

Упомянуть существующий модуль [`composeApp/src/commonMain/kotlin/ru/towich/achline/App.kt`](composeApp/src/commonMain/kotlin/ru/towich/achline/App.kt) как текущую заготовку UI — целевой экран заменит/дополнит его.

### 7. Вне скоупа первой версии (опциональный подраздел)

- Push, синхронизация между устройствами, импорт с рабочего стола в рантайме, тесты с вопросами из сети.

### 8. Критерии приёмки (кратко)

- Оффлайн работа; сохранение прогресса после перезапуска; корректный merge бандл + user; соответствие правилам выбора темы/вопроса; add/delete через FAB; 3 карточки и свайпы.

## Что останется на ваше короткое решение перед финализацией ТЗ

Один заход: подтвердить **tie-breaker** при `correctCount == 0` у многих вопросов (рекомендация в документе — учёт `shownCount`) и **поведение удаления** (текущая карточка vs список). Их можно вынести в подраздел «Решения» в самом md после вашего ответа.
